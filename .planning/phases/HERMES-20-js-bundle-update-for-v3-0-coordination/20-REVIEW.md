## Cross-repo invariant check

The most consequential review angle for Phase 20 is the cross-repo
JID-handling invariant. Verified BYTE-IDENTICAL on the relevant
substring:

```
Python  src/chatlytics_hermes/tools.py:223
        r"^.+@(c\.us|g\.us|lid|newsletter)$"
                ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

JS      servers/chatlytics-mcp.js:60
        /@(c\.us|g\.us|lid|newsletter)$/i
           ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
```

Differences (BENIGN, documented in the Python source comment at
`tools.py:200-206`):

1. **Anchor:** Python is `^.+` explicit; JS is `^` implicit via
   `.test()` semantics. Both require a non-empty prefix before
   the `@` (JS enforces non-empty via the `s.length === 0` guard
   at line 59).
2. **Case:** Python is case-sensitive; JS is `/i` case-insensitive.
   Real-world WAHA JIDs are lowercase, so case never differentiates
   valid inputs. The case dimension is a strict superset on the JS
   side — any valid Python input is also valid JS input.

No behavioral divergence on the well-formed-JID acceptance set. The
JS side accepts a slightly larger set (uppercase suffix variants) but
those never appear in production WAHA traffic.
