# v2.0 Milestone-Wide Code Review

This review is intentionally **orthogonal** to the per-phase reviews under
`.planning/milestones/v2.0-phases/HERMES-0[1-6]-*/0?-REVIEW.md`. It treats
v2.0 as a single shipped artifact and looks for issues that only surface
when the six phases are read together against the live upstream contract
(`/tmp/hermes-ref-v0.14.0/`).

The headline finding (BL-01) was anticipated as a MED in 04-REVIEW
(`04-MED-01`, `_keep_typing` async-cm shape divergence) but is being
**re-classified as a BLOCKER** here. The per-phase review concluded
"all callers are in-plugin" — that is false. The upstream base class
calls the override on every inbound message via
`asyncio.create_task(self._keep_typing(chat_id, metadata=...))`, and
**both** the `metadata` kwarg AND the asynccontextmanager return shape
will crash the path. The reason the test suite is green is that
`tests/test_inbound.py` replaces `adapter.handle_message` with a recorder
before any base-class path runs (see `_install_recorder`), and
`scripts/smoke.sh` never starts a real Hermes gateway (06-MED-01). The
production loader will hit this on the first inbound message.

## 1. Cross-phase consistency

### MD-01 (Medium) — Three subtly different success-shape coercions

**Files:**
- `src/chatlytics_hermes/adapter.py:343-362` (`send` and `_make_send_result`)
- `src/chatlytics_hermes/adapter.py:890-909` (`_standalone_send`)
- `src/chatlytics_hermes/tools.py:58-71` (`_ok` helper)

**Issue:** All three sites implement "if HTTP 200 and payload.success is
truthy-or-absent, return success=True." They each re-derive the same
predicate independently with slightly different defaults:

- `adapter._make_send_result`: `payload.get("success", True)` — absent
  key → True.
- `_standalone_send`: same predicate, then `result.update(payload);
  result["success"] = True` to force-override a payload-side success
  flag.
- `tools._ok`: assumes the caller already filtered status; coerces by
  spreading payload then re-asserting `out["success"] = True`.

Today these produce the same result for the documented Chatlytics
contract. But if the gateway ever returns `200 {"success": false, "error":
"throttled"}`, `adapter.send` correctly fails (it checks success before
returning True), `tools._post` correctly fails (it inspects status code
first via `_err_from_response` on 4xx, then `_ok` on 2xx — but `_ok`
will **lie** and return success=True for a 200+false payload because the
final `out["success"] = True` overrides), and `_standalone_send` also
lies. The same gateway contract is interpreted three different ways.

**Fix:** Centralize the success-coercion into a single helper used by
all three sites:

```python
# In a new src/chatlytics_hermes/_result.py or extend client.py
def coerce_success(status_code: int, payload: Any) -> tuple[bool, Optional[str]]:
    """Returns (success, error_msg)."""
    if status_code >= 400:
        err = (payload.get("error") if isinstance(payload, dict) else None) \
              or f"HTTP {status_code}"
        return False, err
    if isinstance(payload, dict) and payload.get("success") is False:
        err = payload.get("error") or "gateway reported success=false"
        return False, err
    return True, None
```

Then `tools._ok`, `adapter._make_send_result`, and `_standalone_send`
all use it. Eliminates the 200+`success:false` divergence.

**Effort:** small.

## 3. Security — holistic

### BL-01 (BLOCKER) — `_keep_typing` async-cm override crashes the inbound dispatch path

**File:** `src/chatlytics_hermes/adapter.py:741-806`

**Issue:** This is the milestone's most serious latent defect. The override
is decorated `@contextlib.asynccontextmanager`, with signature
`(self, chat_id: str, interval: float = 30.0)`.

Upstream `BasePlatformAdapter` (in
`/tmp/hermes-ref-v0.14.0/gateway/platforms/base.py:3045-3057`) invokes
`_keep_typing` like this on every inbound message:

```python
_keep_typing_kwargs = {"metadata": _thread_metadata}
try:
    _keep_typing_sig = inspect.signature(self._keep_typing)
except (TypeError, ValueError):
    _keep_typing_sig = None
if _keep_typing_sig is None or "stop_event" in _keep_typing_sig.parameters:
    _keep_typing_kwargs["stop_event"] = interrupt_event
typing_task = asyncio.create_task(
    self._keep_typing(
        event.source.chat_id,
        **_keep_typing_kwargs,
    )
)
```

Two compounding failures hit when the live Hermes gateway dispatches an
inbound message to a Chatlytics adapter:

1. **`metadata` kwarg crash.** The base path ALWAYS passes
   `metadata=_thread_metadata` (no signature gate). The chatlytics
   override doesn't accept `metadata`. Python raises `TypeError:
   _keep_typing() got an unexpected keyword argument 'metadata'`.

2. **`asyncio.create_task` rejects the return value.** Even if the
   `metadata` problem were fixed, the asynccontextmanager-decorated
   method does NOT return a coroutine when called — it returns an
   `_AsyncGeneratorContextManager`. `asyncio.create_task(<that>)` raises
   `TypeError: a coroutine was expected, got <_AsyncGeneratorContextManager>`.

**Why the test suite is green:** `tests/test_inbound.py:98-106`
(`_install_recorder`) overwrites `adapter.handle_message` with a
list-recorder before any inbound test runs. The base class's full
`handle_message` (which is what calls `_keep_typing`) is never invoked
under pytest. The recorder pattern is correct for asserting "the
adapter dispatched", but it bypasses the real `handle_message`
pipeline. `scripts/smoke.sh` (06-REVIEW MED-01) also never starts the
gateway, so neither test layer catches this.

**Why the per-phase review under-rated this:** 04-MED-01 noted "shape
divergence" but the reviewer concluded "All callers today are in-plugin."
That is not true. The `_keep_typing` method is the upstream base class
contract — the base class IS a caller, on every inbound message. This
should have been HIGH at minimum at the time. With the milestone-wide
view + verification of upstream call site, it is BLOCKER.

**Impact:** First production inbound message crashes the
`handle_message` task chain. Hermes-side this is caught at the outermost
gateway message-loop boundary (so the gateway process doesn't die) but
NO message is ever processed by the agent. The Chatlytics adapter is
DOA for inbound traffic in production.

**Fix:** Two acceptable shapes:

**Option A (faithful to upstream contract):** Rewrite as a plain
coroutine matching the base signature, with the chatlytics-specific
heartbeat semantics inside:

```python
async def _keep_typing(
    self,
    chat_id: str,
    interval: float = 30.0,
    metadata: Optional[Dict[str, Any]] = None,
    stop_event: Optional[asyncio.Event] = None,
) -> None:
    """Continuously refresh the typing bubble until cancelled.

    Compatible with BasePlatformAdapter._keep_typing call-site in
    base.py:3053 — accepts metadata and stop_event for forward-compat.
    """
    # Initial fire so the bubble appears immediately.
    try:
        await self.send_typing(chat_id, duration=30.0)
    except Exception:
        logger.debug("send_typing initial fire raised", exc_info=True)

    while True:
        if stop_event is not None and stop_event.is_set():
            return
        try:
            await asyncio.sleep(interval)
        except asyncio.CancelledError:
            return
        try:
            await self.send_typing(chat_id, duration=30.0)
        except asyncio.CancelledError:
            return
        except Exception:
            logger.debug("send_typing heartbeat raised", exc_info=True)
```

Then rename the existing `@asynccontextmanager` flavor to something like
`_typing_scope` (keep the convenience wrapper for in-plugin tool
handlers that want `async with`-style scoping):

```python
@contextlib.asynccontextmanager
async def _typing_scope(self, chat_id: str, interval: float = 30.0):
    """In-plugin convenience wrapper around _keep_typing for tool handlers."""
    stop = asyncio.Event()
    task = asyncio.create_task(self._keep_typing(chat_id, interval=interval, stop_event=stop))
    try:
        yield
    finally:
        stop.set()
        task.cancel()
        try:
            await task
        except asyncio.CancelledError:
            pass
```

**Option B (in-place override that doesn't break upstream):** Keep the
asynccontextmanager as `_keep_typing_scope` and DO NOT override the base
`_keep_typing` at all — let the base default coroutine run as-is. The
base default is a no-op (per `base.py:1833`, `send_typing` default is
`pass`), so it would mean losing the heartbeat for inbound-driven
sessions, but the gateway path wouldn't crash. Tool handlers continue to
use `async with adapter._keep_typing_scope(...)`.

**Effort:** small (Option A is the right answer — ~30 LOC, plus update
the test in `test_media.py:296` to use the new context-manager name).

**Acceptance:** Add a new test that calls `adapter._keep_typing(chat_id,
metadata={}, stop_event=asyncio.Event())` directly (no async-with) and
asserts it returns a coroutine, runs, and respects `stop_event.set()`.

### HI-01 (High) — Tool surface exposes arbitrary local file read primitive

**Files:**
- `src/chatlytics_hermes/tools.py:611-705` (all 5 media tools)
- `src/chatlytics_hermes/adapter.py:434-501` (`_resolve_media_url`)

**Issue:** Every media tool (`chatlytics_send_image`,
`chatlytics_send_voice`, `chatlytics_send_video`, `chatlytics_send_file`,
`chatlytics_send_animation`) accepts a `filePath: str` parameter that
becomes the `resource` argument to `_resolve_media_url`. The adapter
then `open(path, "rb")` and uploads the bytes to Chatlytics.

There is zero path validation. An LLM-generated tool call —
or a prompt-injected user message that goads the agent into calling
`chatlytics_send_file(chatId="120...", filePath="/etc/passwd")` — will
read `/etc/passwd` and POST its contents to the configured Chatlytics
gateway, which will then make that file fetchable as a public URL
attached to a WhatsApp message in the attacker's chat.

This is a real-world privilege escalation. The threat model for a Hermes
plugin must assume the LLM is at least partially adversary-controlled
(prompt injection from inbound webhook payloads, untrusted website
content fed to the model, etc.).

**Fix:** Add an allowlist of acceptable upload paths. Two options:

**Option A (env-configured allowlist):**
```python
# In adapter.__init__:
self.upload_allowed_roots: list[Path] = [
    Path(p).resolve()
    for p in (os.getenv("CHATLYTICS_UPLOAD_ALLOWED_ROOTS") or "/tmp").split(os.pathsep)
]

# In _resolve_media_url, local-file branch:
resolved = Path(path).resolve()
if not any(
    str(resolved).startswith(str(root) + os.sep) or str(resolved) == str(root)
    for root in self.upload_allowed_roots
):
    raise PermissionError(
        f"Refusing to upload file outside allowed roots: {resolved}"
    )
```

**Option B (mark tools as "trusted-only"):** Use Hermes's tool
permission framework (if v0.14 exposes one) to require explicit user
consent before each file-upload call. Less surgical but more flexible.

**Effort:** small (Option A is ~10 LOC + 1 env var + README note +
1 test).

**Acceptance:** Add a test that calls
`chatlytics_send_image(chatId=..., filePath="/etc/passwd")` and asserts
`{"success": False, "error": "..."}` is returned WITHOUT touching the
file or making any upload call.

### MD-03 (Medium) — `connect()` health check accepts any 200 response, not just JSON

**File:** `src/chatlytics_hermes/adapter.py:192-209`

**Issue:** The health check accepts any 200 status code from the
configured `base_url + /health`. If a Chatlytics base URL is
misconfigured (typo, internal port pointed at a generic web server),
the adapter happily "connects" against the wrong endpoint. Subsequent
`send()` calls will then hit the wrong server with the configured
Bearer token, leaking the API key in the Authorization header.

**Fix:** Validate the response body is JSON and contains a known
Chatlytics health marker (e.g. `{"status": "ok"}` or `{"service":
"chatlytics-gateway"}`). The Chatlytics gateway's own health response
shape can be confirmed with the gateway team, but at minimum require
JSON parseability:

```python
try:
    health_payload = response.json()
except Exception:
    await self._client.aclose()
    self._client = None
    raise ChatlyticsConnectError(
        f"Chatlytics health endpoint returned non-JSON ({response.text[:200]}); "
        "verify CHATLYTICS_BASE_URL points at a Chatlytics gateway, "
        "not a generic web server."
    )
```

**Effort:** trivial. Material because the Bearer header leak below this
is non-recoverable (api_key is in every subsequent send).

### LO-04 (Low) — `verify_hmac` docstring claims more than the code provides

**File:** `src/chatlytics_hermes/inbound.py:148-163`

**Issue:** Docstring says "Returns False on any malformed input (missing
signature, wrong length, non-hex characters)." `hmac.compare_digest` is
a byte-comparison; it returns False on length mismatch but does NOT
"validate" hex-ness. A signature of `"hello-world"` (11 chars, non-hex)
returns False because length != 64, not because the validator rejected
non-hex. The behavior is fine; the docstring overstates the validation.
Also, the `try/except (TypeError, ValueError)` around `compare_digest`
is unreachable — `compare_digest` raises only on type mismatch
(`bytes` vs `str`), and both arguments here are already strings.

**Fix:** Trim docstring to match behavior:
```python
"""... Returns False on length mismatch or signature mismatch."""
```
And drop the unreachable try/except. Effort: trivial.

### LO-05 (Low) — No replay-attack defense on webhook (no nonce / timestamp window)

**File:** `src/chatlytics_hermes/inbound.py:174-212`

**Issue:** HMAC verification proves authenticity (the sender knows the
shared secret) but not freshness. An attacker who once captures a valid
(payload, signature) pair (e.g. via a TLS-MITM at an earlier point,
network logs, mirror appliance) can replay it indefinitely. WhatsApp
gateway webhooks typically include a `X-Webhook-Timestamp` or similar
that should be verified to be within a small window AND incorporated
into the HMAC pre-image.

**Fix:** Once the Chatlytics gateway team commits to a timestamp header,
add:
```python
ts = request.headers.get("X-Chatlytics-Timestamp")
if not ts or abs(int(ts) - int(time.time())) > 300:  # 5min window
    return web.json_response({"success": False, "error": "stale timestamp"}, status=401)
# And the HMAC pre-image becomes f"{ts}.{body_bytes}" — both sides update.
```

Today the gateway doesn't appear to send a timestamp header (none is
referenced in the code or tests), so this is a v2.1 hardening item once
the upstream gateway exposes one. Mark as LOW for now.

**Effort:** medium (requires gateway-team coordination).

## 4. Resource lifecycle

### LO-06 (Low) — `connect()`/`disconnect()` race window between checking `self._client` and assigning

**File:** `src/chatlytics_hermes/adapter.py:182-261`

**Issue:** `connect()` does `if self._client is None: self._client =
ChatlyticsClient(...)`. If two coroutines `await adapter.connect()`
concurrently on the same event loop, both can pass the `is None` check
before either has assigned, resulting in two `ChatlyticsClient`
instances created (the first one is orphaned without being closed) and
two attempts to bind the aiohttp port (the second raises OSError).

In practice this only matters if the gateway ever calls `connect()`
multiple times concurrently. Upstream `gateway/run.py` appears to call
it serially, so this is theoretical. But the doctrine "connect()/
disconnect() should be idempotent" the codebase advertises is only
half-true — it's idempotent for *sequential* re-entry, not concurrent.

**Fix:** Wrap `connect()` and `disconnect()` body in `async with
self._lifecycle_lock:` (an `asyncio.Lock` instantiated in `__init__`).

**Effort:** trivial. Defense-in-depth; mark LOW.

### IN-01 (Info) — `disconnect()` does not cancel `_keep_typing` tasks spawned by tool handlers

**File:** `src/chatlytics_hermes/adapter.py:263-285`

**Issue:** Tool handlers that use `async with adapter._keep_typing(...)`
spawn a background task. If `disconnect()` is called mid-flight, the
adapter clears `self._client` while the background task is still
sleeping. On the next wakeup, `send_typing()` early-returns
(`self._client is None`), so no crash. But the task is never explicitly
cancelled, just garbage-orphaned. Once GC'd by the loop, the
`asyncio.create_task` warning "Task was destroyed but it is pending"
may surface in logs.

**Fix:** Track outstanding `_keep_typing` tasks in
`self._typing_tasks: set[asyncio.Task]` and `await asyncio.gather(...,
return_exceptions=True)` on them in `disconnect()`. Marginal value;
mark INFO.

## 7. Test coverage gaps

### MD-04 (Medium) — Inbound test harness bypasses base `handle_message`, hiding BL-01

**File:** `tests/test_inbound.py:98-106` (`_install_recorder`)

**Issue:** This is the structural reason BL-01 went undetected. Every
inbound test replaces `adapter.handle_message` with a list-recorder
before exercising the webhook. The recorder catches the
`MessageEvent` and stops there — the base class's full message-handling
pipeline (which is what calls `_keep_typing`) is never invoked.

The recorder pattern is reasonable for asserting "the adapter decoded
the webhook and called handle_message", but it leaves the entire
message-processing path untested. Combined with the smoke
script's lack of a live gateway loader (06-MED-01), the project has no
test that exercises the base path.

**Fix:** Add ONE integration-style test (can stay in the same file)
that does NOT replace `handle_message`. Instead, install a no-op

### MD-05 (Medium) — No test for `connect()` when aiohttp bind fails on port-in-use

### LO-07 (Low) — No test for Chatlytics gateway returning `200 {"success": false}`

**File:** `tests/test_outbound.py`

**Issue:** All `send` tests use `200 {"success": True, ...}`. The
adapter has a code path for `200 + success: false` (see
`adapter.py:343-347` — the `payload.get("success", True)` predicate),
but no test asserts the right behavior. This is exactly the case where
MD-01 above would surface; the missing test is why it's been
unobserved.

**Fix:** Add a test:
```python
async def test_send_returns_false_on_200_with_success_false(adapter, mock_router):
    mock_router.get("/health").mock(return_value=httpx.Response(200, json={}))
    mock_router.post("/api/v1/send").mock(
        return_value=httpx.Response(200, json={"success": False, "error": "throttled"})
    )
    await adapter.connect()
    result = await adapter.send(CHAT_ID, "hi")
    assert result.success is False
    assert "throttled" in result.error
    await adapter.disconnect()
```

**Effort:** trivial.

## 9. Operational readiness

### MD-06 (Medium) — `scripts/smoke.sh` installs hermes-agent on every run with no caching

**File:** `scripts/smoke.sh:36-37`

**Issue:** Every smoke run does `pip install --no-cache-dir
hermes-agent@git+...@v2026.5.16`. On a typical
`python:3.13-slim` cold container, this fetches a hermes-agent
checkout (~30-60 MB) over the network and resolves its full transitive
dep graph (~150 wheels). At ~45-60 s per smoke run (per 06-LOW-01 in
the audit), the smoke pipeline is slow AND non-deterministic — if
`v2026.5.16` is ever GC'd from GitHub, retagged, or moved, smoke
silently uses a different version than yesterday.

**Fix:** Either (a) pin a commit SHA instead of a tag (immutable), or
(b) build a pre-baked docker image (`omernesh/hermes-smoke-base:v0.14.0`)
that contains the hermes-agent install, and have `smoke.sh` pull it
once. Option (a) is one-line. Option (b) is medium effort but pays
back the 45 s every CI run.

**Effort:** small for option A, medium for option B.

### LO-10 (Low) — `_standalone_send` logs nothing on success or failure

**File:** `src/chatlytics_hermes/adapter.py:835-909`

should emit at least one `logger.info` on success ("cron delivered to
{home_channel}: messageId={...}") and one `logger.warning` on failure.

**Fix:** Add structured logs. Effort: trivial.

## Verdict

**FIX_FIRST.**

Per-phase reviews graded each slice individually and all passed with
minors. Read together, the milestone has one production-breaking defect
(BL-01) that the per-phase test harnesses can't see because they all
short-circuit the base-class message-handling pipeline. The dockerized
smoke also doesn't exercise the live loader, so neither gating mechanism
caught it. v2.0.0 as currently tagged will fail on the first real
inbound webhook in production.

The other two HIGH findings (HI-01 arbitrary-file-read via tool surface,
HI-02 contract drift) are also serious. HI-01 in particular is a real
privilege escalation: any prompt-injection or compromised LLM context
gets a `cat /etc/passwd | curl chatlytics-gateway` primitive. HI-03 is
forward-compat brittleness; survivable but worth fixing in the same
pass.

Once BL-01 + HI-01 + HI-02 + HI-03 are fixed, v2.0 ships. The mediums
and lows are all acceptable v2.1 tech debt and individually small.

## Top 3 priorities

If the operator only fixes 3 things before pushing v2.0 to PyPI:

1. **BL-01** — Rewrite `_keep_typing` to honor the base coroutine
   contract (Option A above). Add a regression test that calls it
   directly with `metadata=` and `stop_event=` kwargs. Until this is
   fixed, every inbound message in production crashes the message
   handler. Effort: small (~30 LOC + 1 test).

2. **HI-01** — Add a path allowlist to the media tool's `filePath`
   parameter. Without this, the v2.0 tool surface is a remote arbitrary
   file read primitive (gated by Chatlytics auth, but still). One env
   var (`CHATLYTICS_UPLOAD_ALLOWED_ROOTS`), ~10 LOC of validation, 1
   negative test. Effort: small.

3. **HI-02 + MD-04 (paired)** — Land the live-loader smoke that 06-MED-01
   recommended. The reason BL-01 stayed hidden for 6 phases is that no
   test ever drives the `register(ctx)` path against a real
   `PluginContext` and watches `handle_message` actually run. Even a
   respx-mocked Chatlytics backend behind a real `hermes
   gateway start`-equivalent would have surfaced BL-01 on the first
   inbound event. Until this exists, future regressions will hide in
   the same blind spot. Effort: medium (new test harness in
   `tests/test_live_loader.py` ~80 LOC; possibly a new v2.1 phase).
