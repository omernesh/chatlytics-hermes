# v2.0 PR-Style Second-Opinion Review

## Top-level impression

This is a well-scoped rewrite that delivers what it set out to. The codebase is
small (~2,250 LOC across four source files), readable, and uses idiomatic
modern Python: `from __future__ import annotations`, `Optional[T]`,
`async/await` correctly, narrow `except` clauses with explicit `# noqa: BLE001`
comments where broad catches are intentional. The 45-test suite is real — it
exercises sockets, multipart uploads, HMAC math, and `asyncio.to_thread`
scheduling — not box-checking. Each commit on the branch is small and traces
back to a numbered ROADMAP phase, which makes the history easy to bisect.

What would keep me from a clean APPROVE is a handful of medium-severity items
that a fresh reviewer would flag: a real route-collision footgun in
`connect()`, a `pip install` step that compiles `hermes-agent` from a `git+`
URL inside an alpine slim container (CI surface area), the upstream-contract
divergence on `_keep_typing` that the audit itself flags as a risk, and a
handful of public-symbol leakages in `plugin.yaml`/docstrings that mention
phase numbers (HERMES-03/04/05) which won't mean anything to a user six
months from now. None of these are merge blockers in my view, but they
warrant a follow-up commit before the v2.0.0 tag is pushed publicly.

Overall I'd APPROVE_WITH_NITS — ship it after touching MED-01 (route
collision), MED-04 (phase-number leaks in user-facing manifest), and the
LOW items at the reviewer's discretion.

## Suggested changes (MEDIUM)

### MED-01 — Webhook route collision when `CHATLYTICS_WEBHOOK_PATH=/health`

**File:** `src/chatlytics_hermes/adapter.py:228-229`

```python
app.router.add_post(self.webhook_path, make_webhook_handler(self))
app.router.add_get("/health", make_health_handler())
```

`webhook_path` is operator-configurable via env (`CHATLYTICS_WEBHOOK_PATH`)
or YAML. If anyone sets it to `/health` (a plausible misconfig — they
might think Chatlytics POSTs health checks here), aiohttp's
`UrlDispatcher` allows the registration silently because the methods
differ (POST vs GET), but the operator now has a webhook delivered to a
path they think is a health endpoint and a health endpoint that won't
reject a curl-test. Worse, if the gateway dispatcher ever changes to a
HEAD/GET probe on the webhook path you get silent traffic mixing.

**Fix:** Validate at `__init__` that `webhook_path != "/health"` and
reject with a clear `ValueError`. (This also closes the existing 03-LOW-01
on missing-leading-slash validation while you're there.)

```python
if not self.webhook_path.startswith("/"):
    raise ValueError(
        f"CHATLYTICS_WEBHOOK_PATH must start with '/'; got {self.webhook_path!r}"
    )
if self.webhook_path == "/health":
    raise ValueError(
        "CHATLYTICS_WEBHOOK_PATH cannot be '/health' (reserved for the "
        "embedded health endpoint)"
    )
```

### MED-02 — `_make_tool_handler` returns a non-async-named function but `_bound` IS async

**File:** `src/chatlytics_hermes/adapter.py:962-979`

`_make_tool_handler` builds `_bound` as `async def` and renames it via
`__name__`/`__qualname__` for nicer tracebacks. Fine. But the wrapper is
returned synchronously without any indication to the caller that it must
be awaited. If a future `register_tool` implementation calls it like
`handler(**kwargs)` (sync) the result is a coroutine that no one awaits
and pytest warns once, then silently leaks the coroutine.

**Fix:** Add an `# returns: Coroutine[Dict[str, Any]]` type hint to the
return signature so consumers (and mypy) catch sync invocation:

```python
def _make_tool_handler(ctx: Any, name: str, handler: Any) -> Callable[..., Awaitable[Dict[str, Any]]]:
```

Bonus: add a `RuntimeWarning` in `_bound` if the test harness `register_tool`
ever drops the coroutine on the floor.

### MED-03 — `scripts/smoke.sh` installs `hermes-agent` from `git+` on every run

**File:** `scripts/smoke.sh:36-37`

```bash
pip install --quiet --no-cache-dir \
    "hermes-agent @ git+https://github.com/NousResearch/hermes-agent.git@v2026.5.16"
```

The audit flagged this as 06-LOW-01 ("~45-60s"), but for CI this is
**flaky-by-design** — a transient GitHub outage during `apt-get
install` or `pip install` becomes a smoke-test failure with no signal
about whether your code is broken. In a GitHub Actions runner with a
~10-minute job budget this is non-trivial.

**Fix (minimal):** Add `--retries 3` to `pip install` and wrap the docker
run in a `timeout 600` so a hung pull surfaces clearly. Long-term: cache
the hermes-agent wheel via `actions/cache@v4` keyed on the tag.

### MED-04 — `plugin.yaml` `optional_env` descriptions leak phase identifiers (`HERMES-03`, `HERMES-04`)

**File:** `plugin.yaml:28, 32, 36`

```yaml
description: "Local port for the aiohttp webhook listener (default: 8765, filled in HERMES-03)"
description: "HMAC-SHA256 shared secret for X-Chatlytics-Signature verification (HERMES-03)"
description: "Default chat_id for cron/notification delivery (HERMES-04)"
```

These strings render in the Hermes config UI. End users have no idea
what `HERMES-03` means. The phase identifiers are useful in commit
messages and CONTEXT docs, not in user-facing strings.

**Fix:** Strip the `(HERMES-NN)` annotations from the three optional_env
descriptions. Same nit applies to several docstrings in `adapter.py`
that mention "HERMES-02 / HERMES-04 wires up..." — those are fine
internally but feel like draft notes that shipped.

## Nits & style (LOW / INFO)

### LOW-01 — `hmac.compare_digest` operand normalization is asymmetric

**File:** `src/chatlytics_hermes/inbound.py:160-161`

```python
return hmac.compare_digest(expected, provided_signature.strip().lower())
```

`expected` is hex-encoded lowercase from `hexdigest()`. Calling
`.lower()` on the incoming side is defensive against `Aa12bc...`-style
input, but `.strip()` accepts leading/trailing whitespace which is
non-standard for a header value. Either trust the wire format and drop
both, or accept the bytes form too (some gateways send `sha256=<hex>`).
Currently silently accepts `"  abc...  "` which an operator might find
surprising.

### LOW-02 — Bare `except Exception` in `_keep_typing._beat` cannot fire

**File:** `src/chatlytics_hermes/adapter.py:766-781`

The outer `try/except asyncio.CancelledError:` wrapping the `while True`
is unreachable because the inner `try` already re-raises
`CancelledError`. Functionally harmless (defense in depth) but a senior
reviewer would call it out as dead-error-handling that obscures the
control flow. Recommend collapsing to a single try/except at the inner
scope.

### LOW-03 — `chatlytics_login` mutates `result` dict in-place at line 769

**File:** `src/chatlytics_hermes/tools.py:765-770`

The dict-comprehension `{k: v for k, v in result.items() if k != "success"}`
makes a shallow copy minus `success`, which is fine — but the function
name implies a login operation and the implementation is actually a
re-shaped `/health` probe. Either rename the schema description or
implement actual login (e.g. POST `/api/v1/login` if it exists). The
audit flags this as 05-LOW-03; surfacing it again because the name is
the contract.

### LOW-04 — Test file `tests/test_outbound.py` asserts `adapter.is_connected is True` but the adapter never sets `is_connected`

**File:** `tests/test_outbound.py:77`, `src/chatlytics_hermes/adapter.py:223`

The adapter sets `self._running = True` in `connect()` (line 223 and
260). `is_connected` presumably comes from the base
`BasePlatformAdapter` as a property reading `self._running`. This works
in practice (the test passes), but it's an implicit contract that
relies on upstream's attribute naming. Worth a one-line comment in the
adapter docstring confirming the base class hook so a future refactor
of `_running` does not silently break `is_connected`.

### LOW-05 — `_resolve_resource` returns `Any`, doc says ``None`` is a programming error

**File:** `src/chatlytics_hermes/tools.py:597-608`

The docstring says `None` is a programming error but the schema's
`anyOf` enforces presence at validation time. The function still
returns `None` if both are falsy. The callers (`chatlytics_send_image`,
etc.) then check `if not resource:` and return an error. That's three
layers of defense for one invariant. Either trust the schema and assert,
or remove the in-handler check and let the adapter raise.

### LOW-06 — Inconsistent media-handler API: `send_image` vs `send_image_file`

**Files:** `src/chatlytics_hermes/adapter.py:605-737`,
`src/chatlytics_hermes/tools.py:611-704`

The adapter splits image sending into `send_image(url)` and
`send_image_file(path)`. The tool layer unifies them into a single
`chatlytics_send_image(mediaUrl | filePath)`. The other four media
handlers (`send_voice/video/document/animation`) accept either shape
via `Union[str, bytes, bytearray]` and dispatch internally based on
prefix. The image API is the only outlier. Either give the others
their `_file` counterparts (no — that bloats the API) or fold
`send_image_file` into `send_image`'s `Union` argument. The tool layer
already does the right thing; the adapter layer should match.

### LOW-07 — `chatlytics_dispatch` accepts a `session` kwarg but the schema docstring doesn't explain semantics

**File:** `src/chatlytics_hermes/tools.py:437`

`"session": {"type": "string", "description": "Optional session ID."}` —
fine, but what's a session ID? Is it the Chatlytics account_id under
another name? An LLM agent invoking this tool from a schema-only
description has no way to know. Either link the Chatlytics docs or
expand the description to "Chatlytics session/account ID — same value
as CHATLYTICS_ACCOUNT_ID env var".

### INFO-01 — `inbound.py:_MEDIA_TYPE_MAP` shadows `adapter.py:_MEDIA_TYPE_MAP`

Two different modules define `_MEDIA_TYPE_MAP` with overlapping but
non-identical semantics. The inbound map translates Chatlytics→Hermes
(`"image"` → `"PHOTO"`), the adapter map translates Hermes-handler-name
→ Chatlytics wire (`"document"` → `"file"`). Names are technically
private to each module so no collision in import space, but a developer
grepping for the wire constants will get two hits. A single
`media_types.py` with both maps named clearly (`INBOUND_TO_HERMES`,
`HANDLER_TO_WIRE`) would be cleaner. Not blocking.

### INFO-02 — `_standalone_send` duplicates a chunk of `send()`'s response-parsing logic

**File:** `src/chatlytics_hermes/adapter.py:835-909`

The 200/`success: True` derivation, the `raw_text` fallback, the error
extraction — all duplicate the `_make_send_result` private method.
Refactor: extract a module-level `_parse_send_response(response)` and
call it from both. Saves ~30 LOC.

### INFO-03 — Two test files assert `len(TOOLS) == 21` (`test_tools.py:169`, `test_tool_schemas.py:110`)

Intentional per the comment in `test_tools.py:13` ("duplicated... so
the count guard fires in either file"), but the docstring at the top of
`test_tools.py:14` says `AC-7: len(TOOLS) >= 13` — which is the OLD
minimum from when the tool surface was smaller. The actual assertion is
`== 21`. Docstring drift. Two-line fix.

### INFO-04 — `_FakePlatformConfig` is copy-pasted across three test files

**Files:** `tests/test_outbound.py:30-46`, `tests/test_inbound.py:39-48`,
`tests/test_media.py:31-40`

Three nearly-identical stub classes. Move to `conftest.py` as a fixture
or shared helper. Tests are already async — this is a `@pytest.fixture`
not a class factory pattern shift.

### INFO-05 — `cron_deliver_env_var` registered as a string env-var NAME, not the value

**File:** `src/chatlytics_hermes/adapter.py:1020`

```python
cron_deliver_env_var="CHATLYTICS_HOME_CHANNEL",
```

This is correct per the canonical Hermes contract (Hermes resolves the
env var at cron time), but a reviewer who hasn't read
`hermes_agent/cron/...` will assume this is a literal channel ID and
flag it. One inline comment ("Hermes resolves this env var at cron
dispatch; do not inline the channel ID here") would save the next
reviewer 30 seconds.

## Test review (the 45-test suite as a whole)

**Verdict: well above bar.** Honest assessment with no padding.

**What's good:**

- **Tests are behavioral, not implementation-mirroring.** `test_inbound.py`
  binds a real ephemeral aiohttp port and drives traffic through
  `httpx.AsyncClient` — it would catch a regression where the route
  factory pattern broke even if the unit-level code looked fine
  (`test_connect_starts_aiohttp_server`, line 217). `test_media.py:233`
  goes further and spies on `builtins.open` to assert the file read
  actually runs off the event loop thread. That's a thoughtful
  regression guard, not a checkbox test.
- **HMAC tests cover both sides.** `test_hmac_verification_rejects_bad_signature`
  uses a 64-hex-char wrong digest (not just `"bad"`) — that catches
  length-only fast-paths. The accept-good-signature test computes the
  signature with the same algorithm the adapter uses against a
  byte-exact body, which is the right contract.
- **`respx.MockRouter.routes`-driven assertions** in `test_all_requests_carry_bearer_auth`
  (line 211) iterate every recorded call rather than asserting only the
  last — that's the correct way to do cross-cutting auth-header tests
  and would catch a regression where one specific endpoint accidentally
  dropped the header.
- **Negative paths exist.** `test_connect_raises_on_non_200_health`,
  `test_send_returns_ok_false_on_400`, `test_standalone_send_returns_error_when_env_unset`
  — failure paths are tested, not just happy paths. Many adapters at this
  size skip negative tests; this PR doesn't.

**What could be tighter:**

- **`test_keep_typing_heartbeats_every_30s`** (`test_media.py:296`) uses
  `interval=0.05` + `asyncio.sleep(0.12)` and asserts `>= 2` calls. This
  is fundamentally timing-dependent. On a hot CI runner you might get
  3-4 calls in 0.12s and the test still passes; on a paused container
  you might get 1 and it fails. The comment explicitly acknowledges
  this ("tolerate scheduling jitter on busy CI runners") but the
  contract under test is "fires immediately + reissues every interval"
  — that contract is better tested with a mocked `asyncio.sleep` and
  a deterministic count, not real wall-clock. This is the only test in
  the suite I'd call out as a future-flake risk.
- **No test for `_resolve_media_url` with bytes input directly.** Every
  media test goes through `send_image_file` or a URL. The `bytes` /
  `bytearray` branch (`adapter.py:457-462`) is reachable only through
  programmatic adapter calls and would be exercised by no test if a
  future change broke it.
- **No test for the `make_webhook_handler` JSON-body error paths**
  (invalid JSON, non-dict payload). `inbound.py:191-199` returns 400 for
  these but no test asserts that contract. Adversarial input to a public
  webhook deserves an explicit test.
- **Test file naming.** `test_register.py` covers HERMES-01 acceptance,
  `test_outbound.py` covers HERMES-02, etc. — this is phase-numbered
  organization, which makes the tests easy to trace today but harder to
  find later. (e.g. "where's the test for the public health endpoint?"
  → not obvious without reading the ROADMAP.) Minor.

**45 tests doing real work.** No single test felt redundant. The
duplicate `len(TOOLS) == 21` assertion is intentional (different module
import paths), and the audit caught it.

  the collision surfaces only when looking at both registrations on the

  embedded in HERMES-03 wouldn't flag "HERMES-03" in the description as
  noise — it was accurate at the time. Six months later it's archaeology.
  Only an end-to-end pass catches the cumulative effect.
- **LOW-06 (inconsistent `send_image` vs `send_image_file` API shape).**
  Adapter and tool layers were reviewed in different phases (04 and 05).
  Per-phase reviewers had no incentive to flag cross-layer inconsistency.
- **INFO-04 (duplicated `_FakePlatformConfig` across 3 test files).**
  Each phase added one. By phase 04 the pattern was established; no
  per-phase reviewer was looking at all three at once.

**What per-phase reviews caught that this PR review confirms:**

- 04-MED-01 (`_keep_typing` async-cm shape divergence) — flagged then,
  still relevant now, documented in README "Architecture notes". I'd

- 02-LOW-02 (`send_typing` log flood at WARNING). Real, deferred,
  acceptable.
- 06-MED-01 (smoke does not invoke real `register(ctx)` against the live

**Net:** PR-style review surfaces ~4 cross-cutting items (route
